# running OpenClaw on Manjaro

This document evaluates OpenClaw deployment options with focus on safety and practical implementation on Manjaro Linux.

## Overview

OpenClaw is a **self-hosted AI gateway** that connects chat apps (Discord, Telegram, WhatsApp, Signal, Slack, iMessage, Matrix, etc.) to AI coding agents. It runs as a single Gateway process on your machine and bridges messaging apps to AI assistants with tool use, sessions, memory, and multi-agent routing.

**WARNING: OpenClaw has been reported to brick machines when given autonomous access to host systems.**

### Key Features

- Multi-channel support (one Gateway serves multiple channels simultaneously)
- Agent-native with tool execution (shell, file operations, browser control, web search)
- Browser interaction via managed Chrome/Chromium profiles
- **Sandboxing support for isolated tool execution (CRITICAL for safety)**
- Self-hosted (runs on your hardware, your rules)
- Open source (MIT licensed)

## Requirements

1. Can't brick the host
2. Selective tool isolation
3. Selective host filesystem access
4. Network control
5. Rist of data loss
6. Recover from Mistakes
7. Setup complexity
8. Browser Access
9. versioning and backup of the stack configuration and OpenClaws internal state and memory (two separate)

## Considered Options

### Option 1: Native Installation without Sandboxing

- ❌ Can brick host (Req 1) - Agent runs `rm -rf ~` with your user permissions
- ❌ No tool isolation (Req 2) - All tools execute directly as your user
- ❌ Full filesystem access (Req 3) - Agent can read/write/delete any file you own
- ❌ No network control (Req 4) - Agent uses your network without restrictions
- ❌ High data loss risk (Req 5) - No barrier between agent and your files
- ❌ No recovery mechanism (Req 6) - Deleted files are gone unless you have separate backups
- ✅ Lowest complexity (Req 7) - Just `npm install && openclaw gateway`
- ✅ All browser modes work (Req 8) - Direct access to host browser without configuration
- ✅ Config: easily backed up from `~/.openclaw/` (Req 9) - Standard directory in your home
- ✅ State: easily backed up from `~/.openclaw/memory/` (Req 9) - Subdirectory of config

### Option 2: Native Installation with Sandboxing

- ✅ Cannot brick host - tools run in Docker containers (Req 1) - Agent's `rm -rf /` only affects throwaway container
- ✅ Per-tool/session isolation available (Req 2) - Each tool execution gets fresh container
- ✅ Configurable workspace access (Req 3) - Can mount specific directories read-only or not at all
- ✅ Network control per sandbox (Req 4) - Can set `network: "none"` or custom networks per sandbox
- ✅ Low risk - only sandbox affected (Req 5) - Host files untouched when agent misbehaves
- ✅ Easy recovery - delete sandbox container (Req 6) - `docker rm` removes failed containers
- ⚠️ Medium complexity - requires Docker setup (Req 7) - Need Docker installed and configured for sandboxing
- ✅ All browser modes work natively on host (Req 8) - Gateway runs on host, direct browser access
- ✅ Config: easily backed up from `~/.openclaw/` (Req 9) - Standard directory in your home
- ✅ State: easily backed up from `~/.openclaw/memory/` (Req 9) - Subdirectory of config

### Option 3: Rootless Docker with Sandboxing

- ✅ Cannot brick host (Req 1) - Gateway and tools both containerized
- ⚠️⚠️ Per-tool isolation - **FAILED in testing** (Req 2) - Docker-in-Docker path mapping broke sandbox creation
- ✅ Configurable access (Req 3) - In theory, same as Option 2 but containerized
- ✅ Network control (Req 4) - Both gateway and sandbox containers can have separate networks
- ✅ Low risk (Req 5) - Multiple layers of isolation
- ✅ Easy recovery (Req 6) - Delete both gateway and sandbox containers
- ❌❌ High complexity - **Docker-in-Docker issues** (Req 7) - Needed subgid extension, socket permissions, path mapping workarounds
- ⚠️ Browser modes require extra setup (Req 8) - Browser on host, gateway in container needs network configuration
- ✅ Config: `~/.openclaw/` but owned by container UID (Req 9) - Volume mount from host but files owned by UID 100999
- ✅ State: `~/.openclaw/memory/` but permission issues (Req 9) - Same permission complexity as config

### Option 4: Rootless Docker without Sandboxing

- ✅ Cannot brick host - tools run in container, not on host (Req 1) - Agent's `rm -rf /` affects container, not host filesystem
- ⚠️ No per-tool isolation - tools share gateway container (Req 2) - One tool can interfere with another or gateway process
- ✅ Host filesystem protected - only `~/.openclaw/` mounted (Req 3) - Agent can't access `/home/user/Documents` or other host paths
- ✅ Container network isolation from host (Req 4) - Container has own network namespace, configurable
- ✅ Host data safe - gateway container disposable with backups (Req 5) - Worst case: restore `~/.openclaw/` and `docker compose up`
- ✅ Easy recovery - `docker compose restart` or restore from backup (Req 6) - Gateway disruption fixed by container restart
- ✅ Medium complexity - **worked in testing** (Req 7) - Standard docker-compose setup without Docker-in-Docker
- ⚠️ Browser setup required (Req 8) - Need to configure browser access from container to host or use remote CDP
- ✅ Config: `~/.openclaw/` mounted volume, easy backup (Req 9) - Volume mount makes host files directly accessible
- ✅ State: `~/.openclaw/memory/` mounted volume, easy backup (Req 9) - Same volume mount, standard backup tools work

## Decision

### Recommended: Option 4 (Rootless Docker without Sandboxing)

Given requirements, backup strategy, and testing results:

**Why Option 4:**

- ✅ Host protected - tools run in container, not on host
- ✅ Simple setup - **already working from our testing**
- ✅ Gateway container is disposable with backups
- ✅ Config/state easily backed up via mounted volumes
- ✅ No Docker-in-Docker complexity (which failed in Option 3)

**Trade-offs:**

- Tools can interfere with gateway inside container (but host stays safe)
- Worst case: `docker compose restart` or restore from backup
