# OpenClaw Deployment Guide for Manjaro

## ⚠️ CRITICAL SAFETY WARNING

**OpenClaw has been reported to brick machines when given autonomous access to host systems.** AI agents can execute arbitrary shell commands, modify files, and perform system operations. Without proper sandboxing, this poses a **severe risk** to system stability and data integrity.

**This guide prioritizes blast radius limitation and safe deployment practices.**

---

## Overview

OpenClaw is a **self-hosted AI gateway** that connects chat apps (Discord, Telegram, WhatsApp, Signal, Slack, iMessage, Matrix, etc.) to AI coding agents. It runs as a single Gateway process on your machine and bridges messaging apps to AI assistants with tool use, sessions, memory, and multi-agent routing.

### Key Features
- Multi-channel support (one Gateway serves multiple channels simultaneously)
- Agent-native with tool execution (shell, file operations, browser control, web search)
- Browser interaction via managed Chrome/Chromium profiles
- **Sandboxing support for isolated tool execution (CRITICAL for safety)**
- Self-hosted (runs on your hardware, your rules)
- Open source (MIT licensed)

### Requirements
- Node.js 24 (recommended) or Node 22.14+ (LTS compatibility)
- API key from a model provider (Anthropic, OpenAI, Google, etc.)
- For browser interaction: Chrome, Brave, Edge, or Chromium
- **Docker (REQUIRED for safe sandboxing)**

### Browser Interaction Capabilities

OpenClaw provides sophisticated browser automation through multiple control modes:

#### 1. **Managed Local Browser** (Default)
- OpenClaw launches dedicated Chrome/Chromium profile
- Isolated from personal browser
- Full control via Chrome DevTools Protocol (CDP)
- Dedicated tabs, snapshots, screenshots, PDFs
- Agent can open tabs, read pages, click, type, navigate

#### 2. **Existing Session** (Chrome MCP)
- Attach to your running Chrome browser
- Uses Chrome DevTools MCP
- Requires user approval prompt
- Access to logged-in sessions

#### 3. **Remote CDP**
- Browser runs elsewhere (another machine/container)
- Control via CDP over network
- Good for sandboxed scenarios

#### 4. **Sandboxed Browser**
- Browser in separate Docker container
- Maximum isolation
- Controlled via network CDP

## Deployment Options

### Option 1: Native Installation ⚠️ DANGEROUS - NOT RECOMMENDED

Runs the Gateway and all tool execution directly on your host system with NO isolation.

**Pros:**
- ✅ Fastest performance (no containerization overhead)
- ✅ Easy debugging of gateway code itself
- ✅ Browser works out-of-the-box (all modes supported)

**Cons (CRITICAL):**
- ❌❌❌ **CAN BRICK YOUR MACHINE** - AI can run any command as your user
- ❌❌❌ Full filesystem access (can delete system files, corrupt data)
- ❌❌❌ No undo for destructive operations
- ❌❌❌ AI mistakes = system damage
- ❌ Can modify shell configs, SSH keys, credentials
- ❌ Can install packages, modify system state

**Browser Support:** All modes (managed local, existing session, remote CDP) work natively.

**When to use:** NEVER on your main system. Only in disposable VMs for gateway debugging.

---

### Option 2: Rootless Docker + Full Sandboxing ✅ RECOMMENDED

Gateway runs in Docker container, tool execution runs in isolated sandbox containers.

**Pros (SAFETY CRITICAL):**
- ✅✅✅ **Cannot brick your host** - Tools run in disposable containers
- ✅✅✅ **Filesystem isolation** - Agent only sees sandbox workspace
- ✅✅✅ **Network control** - Can be configured per use case (none/bridge/custom)
- ✅ Rootless mode: No root privileges needed
- ✅ Per-session or per-agent sandboxing (fresh container each time)
- ✅ Read-only filesystem options (agent can't persist malware)
- ✅ Resource limits (prevent resource exhaustion)
- ✅ Easy cleanup (just delete container)
- ✅ Mistakes are contained and recoverable

**Cons:**
- ⚠️ More complex setup (~30-45 minutes first time)
- ⚠️ Performance overhead (acceptable tradeoff for safety)
- ⚠️ Requires Docker knowledge
- ⚠️ More disk space (~2-3 GB for images)

**Security Benefits:**
- Agent can't see or modify your home directory (unless explicitly allowed)
- Agent can't access your SSH keys, credentials, or personal data
- Agent can't install packages on your host
- Network access is controllable (can be disabled, or limited to specific networks)
- Even if agent tries `rm -rf /`, only the sandbox container is affected
- Container gets destroyed after session, no persistence of malicious changes

**Network Options:**
- `network: "none"` - No network access (maximum security, but tools can't fetch web resources)
- `network: "bridge"` - Standard Docker bridge (internet + home network access)
- `network: "custom-network-name"` - Custom Docker network (for fine-grained control)
  - Can restrict to internet-only (no home network)
  - Can limit to specific subnets
  - Requires creating the network first
- **For most users:** Use `"bridge"` to allow internet/API access while maintaining container isolation
- **For internet-only access:** Create a custom network with no route to home network (see below)

**Browser Support:**
- ✅ Sandboxed browser in separate container (maximum isolation)
- ✅ Host browser with sandboxed tools (better performance)
- ✅ Remote CDP (for headless scenarios)
- ⚠️ Existing session (Chrome MCP) - requires extra networking setup

---

### Option 3: Docker without Sandboxing ⚠️ NOT RECOMMENDED

Gateway runs in Docker, but tool execution happens inside the gateway container (not isolated).

**Pros:**
- ✅ Gateway isolated from host
- ✅ Standard Docker workflow

**Cons (SAFETY ISSUES):**
- ❌❌ **Tools can still damage the gateway container**
- ❌❌ **Agent can disrupt its own operation** (break the gateway process)
- ❌ Tools share gateway container namespace
- ❌ Agent can fill container disk, break dependencies

**Browser Support:**
- ⚠️ Host browser with non-isolated tools (risky)
- ✅ Remote CDP

**When to use:** Only if you understand it's a compromise and accept potential gateway disruption.

## Complete Setup - Rootless Docker + Full Sandboxing

This is the ONLY safe approach for regular use.

📚 **Full documentation:** https://docs.openclaw.ai/install/docker

### 1. Install Rootless Docker

```bash
# Install rootless Docker
curl -fsSL https://get.docker.com/rootless | sh

# Add to PATH
export PATH=/home/$USER/bin:$PATH
echo 'export PATH=/home/$USER/bin:$PATH' >> ~/.bashrc

# Verify rootless setup
docker info | grep rootless
# Should show: rootless: true
```

📚 **Rootless Docker docs:** https://docs.docker.com/engine/security/rootless/

### 2. Clone OpenClaw Repository

```bash
git clone https://github.com/openclaw/openclaw.git
cd openclaw
```

### 3. Setup OpenClaw with Sandbox Enabled

```bash
# Enable sandbox during setup
export OPENCLAW_SANDBOX=1
export OPENCLAW_DOCKER_SOCKET=/run/user/$(id -u)/docker.sock
./scripts/docker/setup.sh

# This will:
# - Build the gateway image
# - Run onboarding
# - Configure for rootless Docker
# - Start the gateway container
```

📚 **Docker setup docs:** https://docs.openclaw.ai/install/docker  
📚 **Onboarding guide:** https://docs.openclaw.ai/start/wizard

### 4. Build Sandbox Images

```bash
# Build the sandbox container image for tool execution
export OPENCLAW_DOCKER_SOCKET=/run/user/$(id -u)/docker.sock
./scripts/sandbox-setup.sh

# Build sandboxed browser image (optional but recommended)
./scripts/sandbox-browser-setup.sh
```

📚 **Sandboxing docs:** https://docs.openclaw.ai/gateway/sandboxing  
📚 **Browser sandboxing:** https://docs.openclaw.ai/tools/browser

### 5. Configure Sandboxing with Network Access

Edit `~/.openclaw/openclaw.json`:

📚 **Configuration reference:** https://docs.openclaw.ai/gateway/configuration

```json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "all",
        "scope": "session",
        "workspaceAccess": "none",
        "backend": "docker",
        "docker": {
          "network": "bridge",
          "readOnlyRoot": true,
          "image": "openclaw-sandbox:bookworm-slim"
        },
        "browser": {
          "autoStart": true,
          "network": "openclaw-sandbox-browser",
          "allowHostControl": false
        }
      }
    }
  },
  "browser": {
    "enabled": true,
    "defaultProfile": "openclaw",
    "headless": false,
    "noSandbox": false
  },
  "channels": {
    "telegram": {
      "allowFrom": ["+1234567890"]
    }
  }
}
```

📚 **Channel setup:** https://docs.openclaw.ai/channels/telegram  
📚 **Channel security:** https://docs.openclaw.ai/channels/pairing

**Network Configuration Explained:**

- **`network: "bridge"`** (Default Docker bridge) - Allows sandbox to access:
  - ✅ Internet (for web fetch, API calls, package downloads)
  - ✅ Home network services (via host or local IPs)
  - ✅ Container still isolated from host filesystem
  - ⚠️ Agent can access everything your Docker host can access

- **`network: "none"`** (Maximum Security) - No network access:
  - ✅ Blocks all data exfiltration
  - ✅ Maximum isolation
  - ❌ Tools can't fetch web resources
  - ❌ Can't access APIs or services
  - Use when: Working with sensitive data, no network tools needed

- **`network: "openclaw-external"`** (Recommended: Internet-only) - Custom network:
  - ✅ Internet access (public APIs, websites)
  - ❌ NO access to home network services
  - ❌ NO access to other Docker containers
  - ✅ Better security boundary
  - **Setup required:** See below

**Setting up Internet-Only Network (Recommended):**

This creates a Docker network that allows internet access but blocks home network:

📚 **Docker networking:** https://docs.docker.com/network/

```bash
# Create custom network for sandbox (internet-only)
docker network create \
  --driver bridge \
  --opt "com.docker.network.bridge.name=openclaw-ext" \
  --opt "com.docker.network.bridge.enable_ip_masquerade=true" \
  --internal=false \
  openclaw-external

# Add firewall rules to block private networks from this network
# (Requires iptables/nftables - adapt for your system)
sudo iptables -I DOCKER-USER -s 172.18.0.0/16 -d 192.168.0.0/16 -j DROP
sudo iptables -I DOCKER-USER -s 172.18.0.0/16 -d 10.0.0.0/8 -j DROP
sudo iptables -I DOCKER-USER -s 172.18.0.0/16 -d 172.16.0.0/12 -j DROP

# Make rules persistent (Manjaro/Arch)
sudo pacman -S iptables-persistent  # or use firewalld
sudo iptables-save > /etc/iptables/rules.v4
```

Then use in config:
```json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "docker": {
          "network": "openclaw-external"
        }
      }
    }
  }
}
```

**Security Trade-offs:**

| Network Mode | Internet | Home Network | Security | Use Case |
|--------------|----------|--------------|----------|----------|
| `none` | ❌ | ❌ | ✅✅✅ Max | Sensitive data, no network needed |
| `openclaw-external` | ✅ | ❌ | ✅✅ High | General use, internet-only |
| `bridge` | ✅ | ✅ | ✅ Medium | Need home network access |

**Mitigation for network-enabled sandboxes:**
- Use channel allowlists (restrict who can send commands)
- Monitor logs (review what agent is doing)
- Use egress filtering (firewall rules to specific domains)
- Review agent actions before they execute (approval workflow)

## Security Considerations

### Defense in Depth - Required Layers

#### Layer 1: Sandboxing (MANDATORY) ✅
```json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "all",
        "scope": "session",
        "workspaceAccess": "none",
        "backend": "docker",
        "docker": {
          "network": "bridge",  // "none" for max security, "bridge" for internet access
          "readOnlyRoot": true
        }
      }
    }
  }
}
```

**Network Mode Selection:**
- Use `"openclaw-external"` (recommended) if you need:
  - Internet access for web fetch, API calls
  - Package installation in sandbox
  - NO access to home network (better security)
- Use `"bridge"` if you need:
  - Internet AND home network services access
  - Use per-agent overrides for selective home network access
- Use `"none"` if you need:
  - Maximum security (no data exfiltration possible)
  - Working with sensitive data
  - No network-dependent tools

**Per-Agent Network Overrides:**
```json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "docker": {
          "network": "openclaw-external"  // Internet-only default
        }
      }
    },
    "list": [
      {
        "id": "homelab-admin",
        "sandbox": {
          "docker": {
            "network": "bridge"  // This agent can access home network
          }
        }
      }
    ]
  }
}
```

This allows you to:
- Keep most agents internet-only (safer)
- Create specific agents for home network tasks
- Route to the appropriate agent based on task

### What Sandboxing Protects

**With proper sandboxing enabled:**
- ❌ Agent CANNOT access your home directory
- ❌ Agent CANNOT see your SSH keys or credentials
- ❌ Agent CANNOT modify system files
- ❌ Agent CANNOT install packages on host
- ❌ Agent CANNOT break your shell configuration
- ❌ Agent CANNOT persist malware across sessions (with readOnlyRoot)
- ✅ Agent CAN only affect its isolated sandbox container
- ✅ Worst case: delete container and start fresh

**Network Access Considerations:**

With `network: "openclaw-external"` (recommended default):
- ✅ Agent CAN access internet (web fetch, APIs, package repos)
- ❌ Agent CANNOT access home network services
- ⚠️ Agent COULD potentially exfiltrate data via internet
- **Mitigation:** Monitor logs, use channel allowlists

With `network: "bridge"` (when home network needed):
- ✅ Agent CAN access internet
- ✅ Agent CAN access home network services (databases, local APIs)
- ⚠️⚠️ Agent COULD access/modify home network devices
- **Mitigation:** Use per-agent overrides, only give to trusted/specific agents

With `network: "none"` (maximum security):
- ✅ Agent CANNOT exfiltrate data
- ✅ Agent CANNOT make outbound connections
- ❌ Agent CANNOT use network-dependent tools
- Use when: Working with sensitive data, no network tools needed

**Best Practice: Multi-Agent Network Strategy**

Use different agents with different network permissions:

```json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "docker": { "network": "openclaw-external" }  // Internet-only default
      }
    },
    "list": [
      {
        "id": "general",
        "name": "General Assistant"
        // Uses default: internet-only
      },
      {
        "id": "homelab",
        "name": "Homelab Manager",
        "sandbox": {
          "docker": { "network": "bridge" }  // Can access home network
        }
      },
      {
        "id": "secure",
        "name": "Secure Assistant",
        "sandbox": {
          "docker": { "network": "none" }  // No network at all
        }
      }
    ]
  }
}
```

Route to agents by:
- **Explicit selection:** `@homelab restart server`
- **Channel routing:** Telegram → general, Discord → homelab
- **Pattern matching:** Commands with "192.168" → homelab agent

📚 **Agent routing:** https://docs.openclaw.ai/channels/channel-routing  
📚 **Multi-agent sandbox:** https://docs.openclaw.ai/tools/multi-agent-sandbox-tools

## Common Issues & Solutions

### Issue: Browser Fails to Start (Linux)

**Problem:** Snap Chromium conflicts
```
Error: Failed to start Chrome CDP on port 18800
```

**Solution:** Install non-snap Chrome:
```bash
sudo snap remove chromium  # If snap installed
sudo pacman -S chromium
# Or: yay -S google-chrome
```

📚 **Browser troubleshooting:** https://docs.openclaw.ai/tools/browser-linux-troubleshooting

### Issue: Rootless Docker Socket Not Found

**Problem:** Socket path wrong for rootless
```
Error: Cannot connect to Docker daemon
```

**Solution:**
```bash
export OPENCLAW_DOCKER_SOCKET=/run/user/$(id -u)/docker.sock
echo "OPENCLAW_DOCKER_SOCKET=/run/user/$(id -u)/docker.sock" >> ~/.openclaw/.env
```

### Issue: Sandbox Container OOM

**Problem:** Not enough RAM for image build (exit 137)

**Solution:**
- Increase Docker memory limit
- Or use pre-built images:
```bash
export OPENCLAW_IMAGE="ghcr.io/openclaw/openclaw:latest"
./scripts/docker/setup.sh
```

📚 **Docker troubleshooting:** https://docs.openclaw.ai/gateway/troubleshooting  
📚 **Pre-built images:** https://github.com/openclaw/openclaw/pkgs/container/openclaw

### Issue: Want Internet Access but NOT Home Network

**Problem:** You want sandbox to access internet but NOT your home network services (better security)

**Solution:**

Create a custom Docker network with firewall rules to block private networks:

```bash
# 1. Create custom network
docker network create \
  --driver bridge \
  --opt "com.docker.network.bridge.name=openclaw-ext" \
  openclaw-external

# 2. Add firewall rules to block RFC1918 private networks
sudo iptables -I DOCKER-USER -s 172.18.0.0/16 -d 192.168.0.0/16 -j DROP
sudo iptables -I DOCKER-USER -s 172.18.0.0/16 -d 10.0.0.0/8 -j DROP
sudo iptables -I DOCKER-USER -s 172.18.0.0/16 -d 172.16.0.0/12 -j DROP

# 3. Make persistent (optional)
sudo iptables-save > /etc/iptables/rules.v4

# 4. Update config
clawdock-cli config set agents.defaults.sandbox.docker.network openclaw-external
clawdock-restart
```

**Verify it works:**
```bash
# Should work (internet):
clawdock-exec curl -I https://google.com

# Should fail (home network):
clawdock-exec curl -I http://192.168.1.1
```

**Note:** Adjust the source subnet (`172.18.0.0/16`) to match your custom network's subnet. Check with:
```bash
docker network inspect openclaw-external | grep Subnet
```

### Issue: Need SPECIFIC Tool/Agent to Access Home Network

**Problem:** You want most tools to use internet-only network, but need ONE specific agent/skill to access home network services.

**Solution: Use per-agent network override**

OpenClaw allows per-agent sandbox configuration. Create a separate agent with `bridge` network while keeping default as `openclaw-external`:

```json
{
  "agents": {
    "defaults": {
      "sandbox": {
        "mode": "all",
        "scope": "session",
        "workspaceAccess": "none",
        "backend": "docker",
        "docker": {
          "network": "openclaw-external",  // Internet-only by default
          "readOnlyRoot": true
        }
      }
    },
    "list": [
      {
        "id": "homelab-admin",
        "name": "Homelab Administrator",
        "sandbox": {
          "docker": {
            "network": "bridge"  // Override: allow home network access
          }
        }
      }
    ]
  }
}
```

**How to use the specific agent:**

In your chat app, you can route to specific agents:
```
@homelab-admin check status of my home server at 192.168.1.50
```

Or configure routing based on session/channel:
```json
{
  "agents": {
    "routing": {
      "byChannel": {
        "telegram": "homelab-admin"  // All Telegram uses home network agent
      }
    }
  }
}
```

**Alternative: Use host networking for specific services**

If you need to access a specific home service occasionally, you can:

1. **Use SSH port forwarding** (most secure):
   ```bash
   # On your machine, forward home service to localhost
   ssh -L 8080:192.168.1.50:8080 localhost
   
   # Then from sandbox (with bridge network), access via host
   curl http://host.docker.internal:8080
   ```

2. **Add specific firewall exception**:
   ```bash
   # Allow access to specific home network IP only
   sudo iptables -I DOCKER-USER -s 172.18.0.0/16 -d 192.168.1.50 -j ACCEPT
   ```
   Add this BEFORE the DROP rules, so specific IPs are allowed through.

3. **Use custom Docker network with selective routing**:
   ```bash
   # Create a second custom network that allows home access
   docker network create openclaw-homelab
   
   # No firewall rules needed - it will have default bridge behavior
   ```
   
   Then assign different agents to different networks:
   ```json
   {
     "agents": {
       "list": [
         {
           "id": "internet-only",
           "sandbox": { "docker": { "network": "openclaw-external" } }
         },
         {
           "id": "homelab",
           "sandbox": { "docker": { "network": "openclaw-homelab" } }
         }
       ]
     }
   }
   ```

**Best practice recommendation:**
- Use internet-only (`openclaw-external`) as default
- Create specific agents with home network access only when needed
- Use channel routing or explicit agent selection to control which agent runs
- Monitor logs for the home network agent more carefully

📚 **Per-agent configuration:** https://docs.openclaw.ai/tools/multi-agent-sandbox-tools  
📚 **Agent routing:** https://docs.openclaw.ai/concepts/multi-agent

### Issue: Sandbox Can't Access Home Network Services

**Problem:** Tools in sandbox can't reach services on your home network (e.g., local databases, APIs)

**Solution:**

If you NEED home network access, use `network: "bridge"`:

```bash
# Update config
clawdock-cli config set agents.defaults.sandbox.docker.network bridge
clawdock-restart
```

**To access host services from sandbox:**
```bash
# In sandbox, use special hostname:
curl http://host.docker.internal:3000

# Or use host IP:
curl http://192.168.1.100:3000
```

**To access other home network devices:**
```bash
# Use their local IP directly:
curl http://192.168.1.50:8080
```

**If still having issues:**

1. **Check Docker network mode:**
   ```bash
   clawdock-cli sandbox explain
   # Should show: network: bridge
   ```

2. **Verify container can reach network:**
   ```bash
   clawdock-exec ping -c 3 192.168.1.1
   ```

3. **Check firewall rules:**
   - Your host firewall may block connections from Docker containers
   - On Manjaro/Arch with firewalld:
     ```bash
     sudo firewall-cmd --zone=public --add-masquerade --permanent
     sudo firewall-cmd --reload
     ```

## 🔒 SUMMARY

**OpenClaw can brick your machine if misconfigured. Follow these rules:**

### ✅ DO:
1. **Use Rootless Docker + Full Sandboxing** (REQUIRED for safety)
2. **Always set `sandbox.mode: "all"`** in configuration
3. **Choose network mode carefully:**
   - Use `docker.network: "openclaw-external"` for internet-only (RECOMMENDED)
   - Use `docker.network: "bridge"` for internet + home network access
   - Use `docker.network: "none"` for maximum security (no network tools)
4. **Use `workspaceAccess: "none"`** or `"ro"` only
5. **Test with `clawdock-cli sandbox explain`** before using
6. **Use channel allowlists** to restrict access
7. **Install ClawDock helpers** for easier management
8. **Monitor logs** when using network access
9. **Set up firewall rules** if using custom network for internet-only access

### ❌ DON'T:
1. **NEVER run native on your main system** (VM only)
2. **NEVER use `sandbox.mode: "off"`** in production
3. **NEVER set `workspaceAccess: "rw"`** without understanding risks
4. **NEVER skip sandboxing** "just for testing"
5. **NEVER trust the AI** to be careful
6. **DON'T blindly enable network** - understand the exfiltration risk

### 🚨 If You Ignore This:
- Your system can be bricked
- Data can be lost permanently
- Configuration files can be corrupted
- Credentials can be exposed
- **You have been warned**
